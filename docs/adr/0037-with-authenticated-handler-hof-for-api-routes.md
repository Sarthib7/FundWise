# `withAuthenticatedHandler` HOF for wallet-session API routes

All wallet-session API routes (~21 today, growing with FW-066…070) wrap their handlers in a single Higher-Order Function at `lib/server/with-authenticated-handler.ts`. The HOF absorbs the session check (`requireAuthenticatedWallet`), the optional rate-limit invocation (`enforceFundWiseRateLimit`), JSON body parsing, the wallet-match assertion (`body[walletField] === session.wallet`), Next.js dynamic-route params awaiting, the try/catch error envelope via `getErrorDetails`, and the `NextResponse.json` wrapping on success. Routes shrink from ~30 lines to ~10 and the auth-bypass risk that lived in per-route wallet-match assertions becomes a single tested helper. The HOF is wallet-session only — Fundy service auth (`requireFundyServiceAuth`) keeps its current shape for the 3 service routes, and public routes (`/api/openapi.json`, `/.well-known/*`) stay raw. A sibling `withServiceHandler` can be added later if Scoped Agent Access work ([ADR-0018](./0018-agent-skill-endpoint-and-fundy-telegram-bot.md)) expands service-auth surface. Body validation stays untyped — callers `as { ... }`-cast — because adding Zod is a separate dependency decision deferred to post-Summit; the HOF's `Body` generic accommodates a future schema option without breaking call sites. Next.js Route Handler middleware was considered and rejected: middleware runs per-request before route resolution, can't easily encode per-route rate-limit keys or body-parsing semantics, and would force a separate per-route inner-handler pattern anyway. The HOF is the right Seam for this codebase.
