# `lib/fees/` Module is deep by design

All FundWise fee surfaces (Creation Fee, Contribution Fee, Reimbursement Fee, Routing Fee at Summit; Holding Fee, Yield Fee, Fiat Deposit Fee post-Summit per [ADR-0032](./0032-fund-mode-take-rate-monetization.md)) live behind a fenced Module at `lib/fees/`. Each fee surface exposes one pure quote function — `Fees.creation`, `Fees.contribution`, `Fees.reimbursement`, `Fees.routing` — that returns a uniform `FeeQuote = { kind, feeAmount, transfers, ledgerRow, display }`. The Module is pure except for one side-effectful operation, `Fees.record({ quote, txSig, cluster })`, which writes the `platform_fee_ledger` row after on-chain success and is the only place the table shape exists. Config (rates, platform fee wallet, USDC mint per cluster) flows in from `getFeeConfig(cluster)` at the caller boundary, so the quote functions stay deterministic and the Module is fully unit-testable with no I/O mocking. Each `FeeQuote.transfers` array includes `createAssociatedTokenAccountIdempotentInstruction` for the Platform Fee Wallet's USDC ATA — first fee per cluster pays ~0.0021 SOL rent for the ATA, every subsequent fee is a no-op; no operator bootstrap script needed. The Module composes with the Squads Module via dependency direction Fee → Squads — Fee builds the fee `TransferLeg`, the caller passes it to `Squads.proposeReimbursement({ platformFee })` (FW-068); Squads never imports Fee. Adding a new fee surface post-Summit is a single new file in `lib/fees/` plus a new entry in `getFeeConfig` — no existing fee code changes. See FW-066 through FW-070 for the rollout.
