# Split `fundwise-mutations.ts` into per-concept Modules

The 2167-line `lib/server/fundwise-mutations.ts` file is replaced by `lib/server/mutations/` with one file per concept: `group`, `member` (Profile folded in — display-name *is* a Member attribute), `expense`, `settlement`, `contribution`, `proposal`, `treasury`, `monetization`. Shared helpers (`getAdmin`, `getGroupOrThrow`, `assertMemberCan`, `assertWalletIsMember`, `isMissingColumnSchemaCacheError`, and friends — 77 references in the old file) live in `lib/server/mutations/_internal.ts` and are not re-exported from the public Module surface. `computeSuggestedReimbursements` moves OUT of mutations to `lib/expense-suggestions.ts` because it's a pure derived-view function used by reads, not a mutation. Callers (~21 API routes) migrate from `import { ... } from "@/lib/server/fundwise-mutations"` to per-concept imports like `import { addExpenseMutation } from "@/lib/server/mutations/expense"` — the import block declares each route's concept dependencies. A transitional barrel (`fundwise-mutations.ts → mutations/index.ts → mutations/*`) keeps callers working through the migration; the barrel is deleted at the end of the API-route migration. Tests mirror the structure — `tests/fundwise-mutations.test.ts` splits into `tests/mutations/{concept}.test.ts`. The split is bundled with the Handler HOF migration ([ADR-0037](./0037-with-authenticated-handler-hof-for-api-routes.md)) because both touch every route's import block — net cost is a single round-trip across the route surface. The 9 concepts don't call each other (per the architecture audit on 2026-05-16); they shared infrastructure, not coupling, so the deletion test concentrates complexity at the per-concept level rather than smearing it across the bag.
