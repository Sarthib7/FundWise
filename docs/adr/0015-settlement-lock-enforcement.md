## Problem

Settlement must be atomic and exactly-once: a given `(group_id, settlement_id)` pair should never be settled twice, and concurrent settlement attempts must fail safely.

## Decision

Settlement lock is enforced **on-chain** by PDA derivation uniqueness and **off-chain** by unique constraint on `(group_id, settlement_id)`:

### On-chain (Anchor)

- Settlement PDA seeds: `["group", group_id.as_bytes(), settlement_id.as_bytes()]`
- Attempting `syscall::create_account` for an already-initialized PDA returns `AccountAlreadyInitialized`
- The program converts this to `SettlementLocked` error
- This guarantees exactly-once settlement per settlement_id

### Off-chain (Supabase)

```sql
CREATE UNIQUE INDEX idx_settlements_unique ON settlements(group_id, settlement_id);
CREATE UNIQUE INDEX idx_settlements_pending ON settlements(group_id, settlement_id) WHERE status = 'pending';
```

The Edge Function performs idempotency check before calling on-chain TX:
```ts
const { data: existing } = await supabase
  .from('settlements')
  .select('id, status')
  .eq('group_id', groupId)
  .eq('settlement_id', settlementId)
  .single();

if (existing && existing.status === 'completed') {
  throw new Error('SettlementAlreadyCompleted');
}
if (existing && existing.status === 'pending') {
  // Replay: retry on-chain TX; do NOT create new row
}
```

### Concurrent Attempt Handling

- **Client A** calls EF → row created with status `pending`
- **Client B** simultaneously calls EF → hits unique constraint → EF returns 409 Conflict
- If Client A's on-chain TX fails, row stays `pending`. Retry path:
  - Query existing pending row
  - Retry on-chain TX (same `settlement_id` → same PDA → `SettlementLocked` if already settled on-chain)
  - On TX success, update existing row to `completed`
  - On TX failure after 3 attempts, mark row `failed`

### Idempotency Key Pattern

`spl_id` = UUID v4 generated by client (not DB auto-increment). Immutable once created.

## Consequences

- Exactly-once settlement semantics across off-chain + on-chain
- Safe concurrent settlement attempts without race conditions
- Retry logic is deterministic: same `spl_id` always maps to same settlement

---

## Alternatives Considered

| Approach | Pros | Cons |
|---|---|---|
| DB row lock (`SELECT FOR UPDATE`) | Simple in Postgres | Unique constraint is more robust across distributed clients |
| Optimistic concurrency (`version` column) | Lightweight | Less explicit than unique business key |
| **Unique `(group_id, settlement_id)` + PDA lock** (chosen) | Explicit domain key; cross-layer enforcement | Requires client to generate UUID (easy) |
