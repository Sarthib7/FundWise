## Findings

### [CRITICAL] FW-CSO-001: Live Supabase RLS exposes private ledger data and allows anonymous write attempts

**Confidence:** 10/10  
**Phase:** Phase 2 - Secrets Archaeology / Phase 9 - Broken Access Control  
**Category:** OWASP A01 / A05 / STRIDE Information Disclosure + Tampering  
**Location:** `supabase/migrations/20260428201405_bootstrap_core_schema.sql:122`, `supabase/schema.sql:239`

**Description:**  
The checked-in Supabase schema grants broad public policies across Groups, Members, Expenses, Expense Splits, Settlements, Contributions, Proposals, and Approvals. This is not only historical scaffolding: an anonymous Supabase REST request with the public publishable key returned live Group rows and Expense rows. A non-mutating insert probe with invalid data reached the database check constraint instead of being denied by RLS, proving anonymous insert policy evaluation is open.

**Exploit Scenario:**  
1. Attacker extracts `NEXT_PUBLIC_SUPABASE_URL` and the publishable key from the deployed frontend.
2. Attacker calls `/rest/v1/groups?select=*` and enumerates private Group IDs, invite codes, names, creators, and treasury metadata.
3. Attacker calls `/rest/v1/expenses`, `/members`, `/settlements`, or related tables to read private ledger and wallet data.
4. Attacker submits direct insert/update requests to tables where policies use `with check (true)` or `using (true)`, bypassing the protected Next.js API routes.

**Evidence:**  
Live anonymous REST probes returned HTTP 200 for Group and Expense reads. The Group probe returned 3 rows with keys `code`, `created_by`, `id`, and `name`; the Expense probe returned 3 rows with keys `amount`, `created_by`, `group_id`, `id`, `memo`, and `payer`. An invalid anonymous insert into `groups` returned database constraint error `23514`, not an RLS denial.

The migration includes open policies such as:

```sql
create policy "Expenses are viewable by everyone"
  on public.expenses for select
  using (true);

create policy "Anyone can create expenses"
  on public.expenses for insert
  with check (true);
```

**Remediation:**  
Immediately replace public ledger policies with member-scoped policies or deny-by-default policies. Since the app now routes protected reads/writes through service-role server handlers, the fastest hotfix is to revoke anon access to ledger tables and keep only narrowly public invite metadata if needed.

Recommended immediate SQL shape:

```sql
drop policy if exists "Expenses are viewable by everyone" on public.expenses;
drop policy if exists "Anyone can create expenses" on public.expenses;
drop policy if exists "Anyone can update expenses" on public.expenses;

create policy "No anon expense access"
  on public.expenses
  as restrictive
  for all
  using (false)
  with check (false);
```

Apply equivalent lock-down to `members`, `expense_splits`, `settlements`, `contributions`, `proposals`, and `proposal_approvals`. If public invite lookup needs to survive, expose it through `/api/groups?code=` only or through a small public view that returns only non-sensitive join metadata.

**Priority:** P0 - fix now

### [HIGH] FW-CSO-002: Expense create/update accepts untrusted ledger amounts and split shares

**Confidence:** 9/10  
**Phase:** Phase 9 - OWASP A04/A05/A06  
**Category:** OWASP A01 / A04 / A06, STRIDE Tampering  
**Location:** `app/api/expenses/route.ts:47`, `app/api/expenses/[expenseId]/route.ts:52`, `lib/server/fundwise-mutations.ts:243`, `supabase/schema.sql:54`

**Description:**  
The protected Expense routes only check that `amount` is a number and `splits` is an array. Server mutations confirm the actor and participants are Group Members, but they do not enforce that Expense amount is a positive integer token amount, every split share is non-negative, or split shares sum exactly to the Expense amount. The database schema also lacks positive/sum constraints, and the update RPC only checks method and non-empty split array.

**Exploit Scenario:**  
1. A valid Group Member creates or edits an Expense through the API, not the UI.
2. They submit `amount: 1` with split shares totaling a much larger value, or negative shares.
3. The server persists the record because membership checks pass.
4. `computeBalancesFromActivity` trusts `expense.amount` and `split.share`, corrupting live Balances and suggested Settlements without corresponding real-world cost or on-chain proof.

**Evidence:**  
Route validation accepts any numeric amount:

```ts
typeof body.amount !== "number" || ... || !Array.isArray(body.splits)
```

Mutation validation only verifies membership before insert:

```ts
await assertWalletsAreMembers(
  data.groupId,
  [data.payer, ...data.splits.map((split) => split.wallet)],
  "Every Expense participant must already be a Member of this Group."
)
```

Balance computation then directly applies the stored values:

```ts
balances[item.data.payer] = (balances[item.data.payer] || 0) + item.data.amount
balances[split.wallet] = (balances[split.wallet] || 0) - split.share
```

**Remediation:**  
Add one shared server-side `validateExpenseLedgerInput` used by create and update before any DB write:

```ts
function validateExpenseLedgerInput(amount: number, splits: { wallet: string; share: number }[]) {
  if (!Number.isSafeInteger(amount) || amount <= 0) {
    throw new FundWiseError("Expense amount must be a positive integer token amount.")
  }

  if (splits.length === 0) {
    throw new FundWiseError("Expense must include at least one split.")
  }

  const total = splits.reduce((sum, split) => {
    if (!Number.isSafeInteger(split.share) || split.share < 0) {
      throw new FundWiseError("Expense split shares must be non-negative integer token amounts.")
    }
    return sum + split.share
  }, 0)

  if (total !== amount) {
    throw new FundWiseError("Expense split shares must add up to the full Expense amount.")
  }
}
```

Add database check constraints for positive `expenses.amount` and non-negative `expense_splits.share`. Keep sum validation in server/RPC logic because it spans rows.

**Priority:** P0 - fix now

### [HIGH] FW-CSO-003: Settlement recording verifies a token transfer but not the live Settlement obligation

**Confidence:** 9/10  
**Phase:** Phase 9 - Insecure Design / Business Logic  
**Category:** OWASP A06, STRIDE Tampering/Repudiation  
**Location:** `app/api/settlements/route.ts:20`, `lib/server/fundwise-mutations.ts:447`, `lib/server/solana-transfer-verification.ts:184`, `lib/expense-engine.ts:65`

**Description:**  
`POST /api/settlements` confirms that the authenticated wallet signed a real SPL token transfer of the claimed amount to another Group Member. It does not recompute the current Group Balance or simplified settlement graph before inserting the Settlement. Any Member can record any real USDC transfer from themselves to any other Member as a Group Settlement, even if they do not currently owe that Member or if the amount exceeds the current settleable edge.

**Exploit Scenario:**  
1. A Group Member sends USDC to another Member for an unrelated reason, or deliberately overpays.
2. They POST the transaction signature as a FundWise Settlement for the Group.
3. On-chain verification passes because the transfer is real and signed by the sender.
4. The ledger treats it as a Settlement, increasing the sender's Balance and decreasing the recipient's Balance, even if no matching debt existed.

**Evidence:**  
`addSettlementMutation` validates mode, integer amount, distinct Members, mint, membership, duplicate transaction, and transfer proof. It does not call `computeBalancesFromActivity` or `simplifySettlements` before inserting:

```ts
await verifySettlementTransfer({
  txSig: data.txSig,
  mint: data.mint,
  fromWallet: data.fromWallet,
  toWallet: data.toWallet,
  amount: data.amount,
})

const { data: settlement, error } = await getAdmin()
  .from("settlements")
  .insert({ ... })
```

Balance computation gives Settlement rows full ledger effect:

```ts
balances[item.data.from_wallet] = (balances[item.data.from_wallet] || 0) + item.data.amount
balances[item.data.to_wallet] = (balances[item.data.to_wallet] || 0) - item.data.amount
```

**Remediation:**  
Before recording the Settlement, fetch the Group snapshot, compute Balances and suggested Settlements, and require an exact current edge match:

```ts
const snapshot = await getGroupDashboardSnapshot(data.groupId, data.fromWallet)
const balances = computeBalancesFromActivity(snapshot.members, snapshot.activity)
const edge = simplifySettlements(balances).find(
  (candidate) =>
    candidate.from === data.fromWallet &&
    candidate.to === data.toWallet &&
    candidate.amount === data.amount
)

if (!edge) {
  throw new FundWiseError("Settlement does not match the current live Group Balance.", 409)
}
```

If partial Settlements are introduced later, make that an explicit product decision and cap `amount <= currentEdge.amount`.

**Priority:** P0 - fix now

### [MEDIUM] FW-CSO-004: Dependency audit reports vulnerable runtime and deploy-tool transitive packages

**Confidence:** 8/10  
**Phase:** Phase 3 - Dependency Supply Chain  
**Category:** OWASP A03  
**Location:** `pnpm-lock.yaml`, `package.json`

**Description:**  
`pnpm audit --audit-level moderate` returned 26 advisories: 11 high, 12 moderate, and 3 low. The most relevant runtime path is `@solana/spl-token -> @solana/buffer-layout-utils -> bigint-buffer@1.1.5` with GHSA-3gc7-fjrx-p6mg. Several high advisories are in `tar` through Vercel build tooling, and several moderate/high advisories are in `undici` through deploy/test tooling.

**Exploit Scenario:**  
For runtime Solana parsing code, a vulnerable transitive parser can become reachable if attacker-controlled binary data reaches affected bigint conversion paths. For deploy-tool dependencies, a compromised or malicious archive/build artifact processed locally or in CI could exploit `tar` path traversal classes.

**Evidence:**  
Audit summary:

```text
26 vulnerabilities found
Severity: 3 low | 12 moderate | 11 high
```

Dependency paths verified with `pnpm why`:

```text
@solana/spl-token 0.4.14 -> @solana/buffer-layout-utils 0.2.0 -> bigint-buffer 1.1.5
vercel 47.0.4 -> @vercel/fun 1.1.6 -> tar 6.2.1
@cloudflare/next-on-pages -> miniflare -> undici 5.29.0
```

**Remediation:**  
Run targeted dependency updates. Start with deploy tooling (`vercel`, `@cloudflare/next-on-pages`, `wrangler`) and evaluate whether `@sqds/multisig` can move off its older `@solana/spl-token` chain. For `bigint-buffer`, no patched version is listed by npm audit, so mitigate by keeping Solana libraries current and avoiding direct parsing of attacker-supplied binary layouts outside trusted SDK paths.

**Priority:** P2 - fix this month

### [LOW] FW-CSO-005: Production security headers are not configured in Next/Cloudflare config

**Confidence:** 8/10  
**Phase:** Phase 9 - Security Misconfiguration  
**Category:** OWASP A02  
**Location:** `next.config.mjs:7`, `wrangler.jsonc:1`

**Description:**  
No application-level security headers were found for CSP, HSTS, frame protection, content-type sniffing, referrer policy, or permissions policy. The current app does not use raw user HTML, so this is defense-in-depth rather than a confirmed exploit, but FundWise is financial software and should reduce browser abuse surface before mainnet-beta.

**Exploit Scenario:**  
If a future XSS, injected third-party script, or compromised dependency lands in the browser path, the lack of CSP and related headers gives the payload fewer browser-enforced constraints. Missing frame controls also leaves more room for clickjacking around wallet-connect or pre-signing screens if the hosting layer does not add equivalent headers.

**Evidence:**  
Searches for `Content-Security-Policy`, `Strict-Transport-Security`, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, and `headers(` returned no app or config matches.

**Remediation:**  
Add headers in `next.config.mjs` or Cloudflare Pages configuration. Start conservative and adjust for wallet adapters:

```js
async headers() {
  return [
    {
      source: "/(.*)",
      headers: [
        { key: "X-Content-Type-Options", value: "nosniff" },
        { key: "X-Frame-Options", value: "DENY" },
        { key: "Referrer-Policy", value: "strict-origin-when-cross-origin" },
        { key: "Permissions-Policy", value: "camera=(), microphone=(), geolocation=()" },
      ],
    },
  ]
}
```

Roll out CSP separately after testing wallet, QR scanner, LI.FI, and Supabase domains.

**Priority:** P3 - backlog / hardening
